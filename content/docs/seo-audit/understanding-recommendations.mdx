## 建議的結構

每個審查建議都包含以下資訊：

| 欄位 | 說明 |
|-----|------|
| **建議標題** | 簡要說明需要做什麼 |
| **問題描述** | 詳細解釋為什麼這是一個問題 |
| **影響範圍** | 這個問題對 SEO 的影響程度 |
| **建議做法** | 具體的修改方式或步驟 |
| **優先級** | 處理的急迫程度（高/中/低） |
| **所屬維度** | AEO / Technical / Content / Competitor |

## 優先級詳解

### 🔴 高優先 (Critical)

這些問題會**嚴重影響** SEO 效果，搜尋引擎可能無法正確理解或索引你的頁面。應該**立即處理**。

**常見的高優先問題：**

| 問題 | 影響 | 建議處理時間 |
|-----|------|-------------|
| 缺少 Title Tag | 搜尋結果顯示異常，嚴重影響點擊率 | 當天 |
| 缺少 Meta Description | 搜尋結果摘要由 Google 自動生成，可能不理想 | 當天 |
| 多個 H1 標籤 | 頁面結構混亂，影響搜尋引擎理解 | 當天 |
| Canonical URL 錯誤 | 可能導致重複內容問題或權重分散 | 當天 |
| 重大 Schema 錯誤 | 結構化資料無效，無法獲得 Rich Results | 1-2 天 |
| 頁面無法爬取 | 搜尋引擎完全無法索引頁面 | 立即 |

### 🟡 中優先 (Important)

這些問題有**明顯改善空間**，處理後能顯著提升 SEO 表現。建議**一週內處理**。

**常見的中優先問題：**

| 問題 | 影響 | 建議處理時間 |
|-----|------|-------------|
| Title 過長/過短 | 搜尋結果顯示可能被截斷 | 一週內 |
| 缺少圖片 Alt 文字 | 影響圖片搜尋排名和無障礙體驗 | 一週內 |
| Heading 結構不完整 | 內容層級不清晰 | 一週內 |
| 缺少 FAQ Schema | 錯失 AI 搜尋引擎的引用機會 | 一週內 |
| 內部連結不足 | 頁面權重傳遞不佳 | 兩週內 |
| Meta Description 過長 | 搜尋結果摘要會被截斷 | 一週內 |

### 🟢 低優先 (Nice to Have)

這些是**進階優化**項目，能讓 SEO 表現更上一層樓，但不處理也不會有明顯負面影響。**有時間再處理**。

**常見的低優先問題：**

| 問題 | 影響 | 建議處理時間 |
|-----|------|-------------|
| 可擴充的 Schema 類型 | 增加 Rich Results 機會 | 視情況 |
| 內容深度可再提升 | 可能提升長尾關鍵字排名 | 視情況 |
| Open Graph 優化 | 改善社群分享預覽 | 視情況 |
| 新增 llms.txt | 引導 AI 爬蟲 | 視情況 |

## 建議類別詳解

### Technical SEO 建議

技術面建議通常修復方式明確，是最容易執行的類型。

| 建議類型 | 問題說明 | 修復方式 |
|---------|---------|---------|
| **Title Tag** | 缺少、過長、過短、重複 | 在 HTML head 或 CMS 設定中修改 |
| **Meta Description** | 缺少、過長、過短、重複 | 在 HTML head 或 CMS 設定中修改 |
| **Canonical URL** | 未設定、設定錯誤、指向不存在頁面 | 在 HTML head 中新增正確的 canonical 標籤 |
| **Heading 結構** | 多個 H1、層級跳躍、缺少 H1 | 調整 HTML 結構，確保只有一個 H1 且層級遞進 |
| **圖片 Alt** | 缺少 Alt 屬性、Alt 為空 | 在 img 標籤中新增描述性的 alt 文字 |
| **Schema Markup** | 格式錯誤、缺少必要欄位 | 修正 JSON-LD 或微資料格式 |

### AEO 建議

AEO (AI Engine Optimization) 建議幫助你的內容更容易被 AI 搜尋引擎理解與引用。

| 建議類型 | 問題說明 | 修復方式 |
|---------|---------|---------|
| **新增 FAQ Schema** | 頁面有 FAQ 內容但沒有結構化資料 | 使用 Otlex 的 FAQ Schema 工具生成 JSON-LD |
| **優化問答格式** | 內容不是以清晰的問答形式呈現 | 將內容改寫為「問題 + 回答」的格式 |
| **提升資訊密度** | 內容冗長但資訊不足 | 精簡冗餘內容，增加有價值的資訊 |
| **新增實體資訊** | 缺少組織、作者、產品等實體標記 | 新增對應的 Schema（Organization、Person、Product） |
| **新增 llms.txt** | 網站沒有 AI 爬蟲指引 | 在根目錄建立 llms.txt 檔案 |

### Content 建議

內容建議關注的是文章本身的品質與 SEO 友善度。

| 建議類型 | 問題說明 | 修復方式 |
|---------|---------|---------|
| **增加內容深度** | 文章過短或資訊不夠完整 | 擴充內容，增加細節、範例、數據 |
| **改善可讀性** | 段落過長、句子複雜 | 分段、使用列表、加入小標題 |
| **優化關鍵字分布** | 關鍵字過於集中或完全沒有 | 自然地在標題、段落中分布關鍵字 |
| **新增內部連結** | 頁面孤立，沒有連到其他相關頁面 | 在內容中加入指向相關頁面的連結 |
| **新增多媒體** | 純文字內容，缺少視覺元素 | 加入相關的圖片、表格、影片 |

### Competitor 建議

競品分析建議幫助你了解與排名靠前的競品之間的差距。

| 建議類型 | 問題說明 | 修復方式 |
|---------|---------|---------|
| **Schema 差距** | 競品有而你沒有的結構化資料 | 新增對應的 Schema 類型 |
| **內容長度差距** | 你的內容明顯比競品短 | 擴充內容（但要確保是有價值的內容） |
| **關鍵字涵蓋差距** | 競品涵蓋了你沒有提到的相關關鍵字 | 檢視是否需要新增這些內容 |

## Local SEO 建議怎麼看（GEO）

當品牌已啟用 Local SEO，審查詳情的「建議」分頁會多一塊 **Local SEO 城市覆蓋率**，並且可用 **GEO** 篩選快速聚焦在地問題。

### 城市覆蓋狀態

| 狀態 | 代表意義 | 建議優先行動 |
|-----|---------|-------------|
| **已覆蓋** | 已有足夠城市訊號（URL / Title-H1 / Schema） | 持續優化高曝光低點擊查詢 |
| **內容過薄** | 有頁面但內容深度或在地訊號不足 | 補足在地內容、FAQ、Schema |
| **缺少頁面** | 找不到可索引的城市頁 | 優先建立城市落地頁與入口連結 |

### 常見 GEO 建議與對應動作

| 建議類型 | 你要做什麼 |
|---------|-----------|
| 缺少城市頁 (`local.missing_city_page`) | 建立「服務 + 城市」落地頁，URL / Title / H1 都要有城市名 |
| 城市頁內容過薄 (`local.thin_city_page`) | 補到至少有足夠在地資訊（服務範圍、案例、FAQ、聯絡方式） |
| 缺 LocalBusiness Schema (`local.missing_local_business_schema`) | 補上 LocalBusiness JSON-LD，含 address / telephone / areaServed |
| 城市修飾詞不足 (`local.missing_city_modifier`) | 在 Title/H1/首段加入自然城市修飾詞 |
| 缺內部連結 (`local.missing_internal_links`) | 從首頁、服務頁、導覽加到城市頁連結 |

<Callout type="info">
  實務上建議先修「缺少頁面」與「Schema 缺失」這兩類，再處理內容深化與 FAQ，通常對在地查詢的改善最快。
</Callout>

## AI 輔助執行功能

部分建議提供一鍵 AI 生成功能，幫助你快速產出優化內容：

### AI 生成 Alt 文字

當圖片缺少 Alt 屬性時：

1. 點擊建議旁的「AI 生成」按鈕
2. Otlex 使用 Vision AI 分析圖片內容
3. 自動生成描述性的 Alt 文字
4. 預覽結果，滿意後複製使用

<Callout type="info">
  AI 生成的 Alt 文字會描述圖片的視覺內容，但可能不了解圖片在文章脈絡中的意義。建議檢查並視需要調整。
</Callout>

### AI 生成 Meta Description

當缺少或需要改善 Meta Description 時：

1. 點擊「AI 生成 Description」
2. AI 分析頁面內容，撰寫符合 SEO 最佳實踐的描述
3. 控制在 150-160 字元，包含目標關鍵字
4. 預覽結果，滿意後複製使用

### AI 生成 FAQ Schema

當頁面適合新增 FAQ 結構化資料時：

1. 點擊「生成 FAQ Schema」
2. AI 從內容中提取問答對
3. 自動格式化為 JSON-LD
4. 複製程式碼貼到頁面的 head 區塊

## 追蹤建議處理進度

### 狀態標記

你可以標記每個建議的處理狀態：

| 狀態 | 說明 |
|-----|------|
| ⏳ **待處理** | 尚未開始處理 |
| 🔄 **處理中** | 正在進行修改 |
| ✅ **已完成** | 已修復並確認 |
| ❌ **忽略** | 決定不處理（需填寫原因） |

### 重新審查驗證

修改完成後，重新執行審查可以：

- 確認問題是否已解決
- 驗證修改是否正確
- 查看分數是否提升

系統會自動對比前後兩次審查結果，顯示：
- 哪些問題已改善
- 哪些問題仍存在
- 是否有新問題產生

## 最佳實踐

### 處理順序建議

1. **先處理高優先問題**：這些問題影響最大
2. **批次處理同類問題**：例如一次修復所有缺少 Alt 的圖片
3. **Technical 問題優先於 Content**：技術問題通常更容易修復
4. **每次修改後重新審查**：確保修改正確

### 避免過度優化

- 不要為了分數而堆砌關鍵字
- Alt 文字要描述圖片，不是塞關鍵字
- Schema 要符合實際內容，不要虛構資訊
